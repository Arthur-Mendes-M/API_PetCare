Sobre o PetCare.

PetCare é a forma simples, organizada e centralizada em um sistema multi-plataforma para te auxiliar na gestão do seu petshop.

Sobre a API

Este é o endpoint raiz da API do PetCare. Aqui você terá um guia completo de como utilizar todos os endpoints da API PetCare connect.

Token de segurança

Por motivos de segurança, foi criado um token secreto que deve ser incluído em TODOS os demais endpoints, como demonstrado na imagem.

O token é lido através do query paramenter (parametro de busca) nomeado auth onde, na url do endpoint a ser utilizado, é necessário específica-lo. Como no exemplo: https://api-petcare.onrender.com/Employees?auth=<TOKEN>

Esse token é único e exclusivamente gerado para o uso desta API de forma interna pelas plafarmas do PetCare.

Portanto, essa API não é pública.

GET

No endpoint dos funcionários, com a rota GET, existem duas opções, onde ambas utilizam da mesma url. O que diferencia as duas é o corpo da requisição (body).

Endpoint

https://api-petcare.onrender.com/Employees?auth=<TOKEN>

Essa é a URL padrão para o endpoint 'Employees'

Buscar todos os funcionários

Corpo da requisição | Nenhum

Nenhum / Não precisa

Basta somente chamar a url padrão para esse endpoint

Resposta da API | JSON / Array

[{   "avatarurl": "link_para_foto",   "name": "nome_do_funcionario",   "email": "email_a_ser_buscado@dominio.com",   "password": "senha_do_funcionario",   "role": "cargo_do_usuario",   "salescount": "quantidade_de_vendas", }]

Obs: o retorno é um ARRAY, logo, para acessar o objeto do funcionário, é necessário acessar o primeiro elemento.

Validar o acesso de um funcionário específico

Corpo da requisição | JSON

{   "email": "email_a_ser_buscado@dominio.com",   "password": "senha_do_funcionario" }

 Obrigatório: formato JSON e os campos: email e password

POST

Na rota POST é necessário que o corpo da requisição (body) seja passado em formato multipart/form-data.

Endpoint

https://api-petcare.onrender.com/Employees?auth=<TOKEN>

Na rota POST a URL se mantém a padrão do endpoint 'Employees'

Cadastrando um funcionário com todos os campos

Corpo da requisição | JSON

{   "image": <arquivo_de_imagem>,   "name": "nome_do_funcionario",   "email": "email_a_ser_buscado@dominio.com",   "password": "senha_do_funcionario",   "role": "cargo_do_usuario",   "salescount": "quantidade_de_vendas", }

Os elementos OBRIGATÓRIOS são somente: "name", "email" e "password".
Resposta da API | JSON / Array

[{   "avatarurl": "link_para_foto" | Null,   "name": "nome_do_funcionario",   "email": "email_a_ser_buscado@dominio.com",   "password": "senha_do_funcionario",   "role": "cargo_do_usuario" | Null,   "salescount": "quantidade_de_vendas" | Null, }]

Obs: o retorno é um ARRAY, logo, para acessar o objeto do funcionário, é necessário acessar o primeiro elemento.

PUT

Na rota PUT é necessário que o corpo da requisição (body) seja passado em formato multipart/form-data com os dados a serem alterados (podendo ser somente o que for alterado), e na url da rota, o id do funcionário a ser atualizado.

Endpoint

https://api-petcare.onrender.com/Employees/<id>?auth=<TOKEN>

Na rota PUT a URL precisa receber o id do funcionário ser atualizado.

Atualizando um funcionário
Corpo da requisição | multipart/form-data

{   "image": <arquivo_de_imagem>,   "email": "email_a_ser_buscado@dominio.com",   "role": "cargo_do_usuario", }

Nessa rota é necessário somente um id existente de funcionário e algum campo com um valor novo.
Resposta da API | JSON / Array

[{   "avatarurl": "link_para_foto" | Null,   "name": "nome_do_funcionario",   "email": "email_a_ser_buscado@dominio.com",   "password": "senha_do_funcionario",   "role": "cargo_do_usuario" | Null,   "salescount": "quantidade_de_vendas" | Null, }]

Obs: o retorno é um ARRAY, logo, para acessar o objeto do funcionário, é necessário acessar o primeiro elemento.

DELETE

Na rota DELETE é necessário somente o id válido de um funcionário existente. Sem a necessidade do corpo de requisição.

Endpoint

https://api-petcare.onrender.com/Employees/<id>?auth=<TOKEN>

Na rota PUT a URL precisa receber o id do funcionário a ser deletado.
